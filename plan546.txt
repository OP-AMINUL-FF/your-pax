4.3 ― ORCHESTRATOR FLOW
─────────────────────────

  your-pax.py (main entry point):
    → Starts orchestrator thread (orchestrator.py)
    → Optionally starts web server thread (webapp.py)
    → Optionally starts display thread (display.py)

  orchestrator.py:
    → Loads actions from config/actions.json
    → Iterates through actions in priority order
    → For each action, checks: target discovered? ports open? not blacklisted?
    → Dispatches action via its b_class.run()
    → Reports results via event_bus.py

  EVENT BUS (event_bus.py):
    → Broadcasts events: system_status, network_scan_progress,
      wifi_handshake_captured, captive_portal_loot, bruteforce_update
    → SSE endpoint (/events) streams these to web UI
    → bt_serial_server.py also reads and forwards events to SPP clients

8. BUG & RISK REPORT
────────────────────────────────────────────────────────────────────────────────

8.1 ― CRITICAL: BT SPP Command Name Mismatches (9 items)
───────────────────────────────────────────────────────────

  DETAIL: The Android app sends 9 commands with names that don't match
  what bt_serial_server.py's CMD_ROUTER expects. These commands will
  receive "unknown_cmd" response instead of the expected data.

  COMMANDS AFFECTED:
    wifi_status         → get_wifi_status         WiFi status page
    bluetooth_status    → get_bluetooth_status     BT connection page
    oneshot             → oneshot_start            WPS attack
    network_data_json   → get_network_data         Network scan results
    netkb_data_json_full→ get_netkb_data           Full knowledge base
    netkb_data_json     → get_netkb_meta           Knowledge base meta
    list_credentials_json→ list_credentials        Credentials page
    restart_your_pax_service→ restart_service      Service restart
    store_data          → get_store_data           Store dashboard

  FIX: Add alias entries in bt_serial_server.py CMD_ROUTER:
    CMD_ROUTER = {
        "wifi_status": handle_wifi_status,      # alias for app compatibility
        "get_wifi_status": handle_wifi_status,  # original
        ...
    }

8.2 ― WARNING: 11 HTTP-Only Commands Unavailable Over SPP
─────────────────────────────────────────────────────────────

  The following 11 commands have NO handler in bt_serial_server.py.
  They work over HTTP but will fail over BT SPP:
    - Service controls: start/stop web/nap/spp
    - Mode switch: switch_mode, get_mode_config
    - Evil AP: stop_evil_clone, wpa_validate_status
    - Bluetooth: bluetooth_devices

  FIX (optional): Add these handlers to bt_serial_server.py if App Only
  mode should support them.

8.3 ― WARNING: Polling Intensity on RPi Zero 2W
──────────────────────────────────────────────────

  /get_logs polled every 1.5 seconds from index.html. On a Zero 2W,
  this means:
    - ~40 file reads/minute from data/logs/temp_log.txt
    - JSON serialization of the entire log file each time
    - Additional overhead from 5 other polling intervals

  RISK: SSD/SD card wear + CPU overhead under load.
  SUGGESTION: Increase /get_logs interval to 3-5s, or add in-memory
  log buffer that only flushes to disk periodically.

8.4 ― WARNING: sdptool Deprecation
─────────────────────────────────────

  sdptool (used in configure_bt_spp.sh) is deprecated in newer bluez
  versions (Debian Bookworm+). The recommended replacement is the D-Bus
  ProfileManager API (org.bluez.ProfileManager1).

  SUGGESTION: Add a fallback in configure_bt_spp.sh that uses Python's
  D-Bus bindings to register SPP if sdptool fails.

8.5 ― WARNING: Bruteforce Thread Count on Zero 2W
─────────────────────────────────────────────────────

  SSH brute-force uses 40 threads (ssh_connector.py). On a 512MB Zero 2W:
    - 40 paramiko SSH connections at once may exceed RAM
    - Each connection uses ~5-10MB → 200-400MB total

  SUGGESTION: Auto-detect available RAM and reduce thread count:
    - RAM < 512MB → use 10 threads
    - RAM < 1024MB → use 20 threads
    - RAM >= 1024MB → use 40 threads

8.6 ― INFO: Comment/Docstring Gaps
─────────────────────────────────────
