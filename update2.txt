1. SYSTEM OVERVIEW
================================================================================

your-pax is an offensive security cyberdeck that runs on a Raspberry Pi.
It boots and:
  - Starts the orchestrator (autonomous attack engine)
  - Starts the web server on port 8000
  - Starts Bluetooth NAP (Network Access Point) with SSID "your-pax"
  - Shows status on e-paper display (if connected)

The phone connects to the Pi via Bluetooth PAN.
Phone gets IP 192.168.4.x from the Pi's Bluetooth NAP.
Pi's Bluetooth NAP bridge IP: 192.168.4.1.
Web server: http://192.168.4.1:8000
Android app connects via REST API to this IP.

Two operation modes:
  A) Autonomous mode — orchestrator continuously scans + attacks targets
  B) Manual mode — user triggers specific attacks via web UI or Android app

Key files:
  your-pax.py        — Entry point, starts orchestrator + webapp + display threads
  orchestrator.py    — The "brain" — orchestrates all actions
  webapp.py          — HTTP server + request routing
  utils.py           — WebUtils class: all endpoint handlers
  shared.py          — SharedData: paths, config, CSV read/write, stats
  config/*.json      — Configuration files
  actions/*.py       — 33 action/attack modules

2. BOOT → BLUETOOTH → CONNECT FLOW
================================================================================

STEP 1: Pi boots → your-pax.py starts (via systemd service your-pax.service)
STEP 2: SharedData.initialize_paths() sets up all directory paths
STEP 3: SharedData.load_config() loads shared_config.json
STEP 4: SharedData.setup_environment() — creates dirs, CSVs, clears logs
STEP 5: E-paper display initialized (if has_display=true)
STEP 6: WebThread starts → web server on port 8000 (multi-threaded)
STEP 7: Orchestrator starts in its own thread:
         - Loads all actions from config/actions.json
         - Runs initial network scan
         - Enters main cycle (scan → execute actions → idle → scan...)
STEP 8: Bluetooth NAP starts (bluetooth_nap.py) — SSID "your-pax"
STEP 9: Phone scans for Bluetooth devices → finds "your-pax"
STEP 10: Phone connects to Bluetooth PAN → gets IP 192.168.4.x
STEP 11: Phone connects to Pi at 192.168.4.1:8000 via REST API
         - Web UI works in browser
         - Android app connects programmatically

Bluetooth NAP details:
  - SSID: "your-pax"
  - Bridge IP: 192.168.4.1
  - Web port: 8000
  - Connected clients: shown in Android app home status bar

5-C. DETAILED ATTACK PARAMETERS
────────────────────────────────

WiFiHandshake — Handshake Capture
  params: bssid (MAC), channel (int), prefix (optional string)
  tool: airodump-ng
  output: .cap files in data/handshakes/
  status: /handshake_status → {"running": bool}

WiFiPMKID — PMKID Capture
  params: bssid (MAC), channel (int), prefix (optional string)
  tool: hcxdumptool
  output: .pcapng files in data/pmkid/
  status: /pmkid_status → {"running": bool}

WiFiDeauth — Deauth Attack
  params: bssid (MAC), client MAC (MAC, default ff:ff:ff:ff:ff:ff = broadcast), count (int 1-10000), channel (optional int)
  tool: aireplay-ng
  notes: count=0 means continuous

WiFiOneShot — WPS Attack
  params: bssid (MAC), pixie (bool), bruteforce (bool), pbc (bool), pin (string), delay (float), pixie_force (bool), show_pixie_cmd (bool), verbose (bool), iface_down (bool)
  tool: OneShot
  status: /oneshot_status → {"running": bool, "output": [line, ...]}

EvilAP — Rogue Access Point
  params:
    ssid (string, default "FreeWiFi")
    channel (int, 1-14)
    password (string, null=open)
    interface (string, default "wlan0")
    portal (string, default "generic.html" — template file in web/portals/)
    wpa_validate (bool) — if true, validates captured password against real AP
    wpa_interface (string)
    karma (bool) — respond to all probe requests
    karma_interface (string)
    clone (bool) — Evil Twin mode (requires 2 interfaces)
    deauth_interface (string)
    clone_interface (string)
    target_bssid (MAC) — AP to clone
    target_ssid (string) — AP SSID to clone
    target_channel (int) — AP channel to clone
  uses: hostapd (AP), dnsmasq (DHCP/DNS), nginx (portal webserver)

EvilClone — Evil Twin (when clone=true in EvilAP)
  - Deauths target AP's clients on deauth_interface
  - Spoofs target AP's SSID/BSSID on clone_interface
  - Runs captive portal
  - Requires 2 Wi-Fi interfaces

CaptivePortal — Phishing Portal
  - Serves fake login page from web/portals/*.html
  - Captures credentials to data/captured_creds.txt
  - Portal templates: generic.html and custom uploads

LootMonitor — Live Monitor on Evil AP
  - Captures DNS queries, HTTP requests, detected devices
  - Data accessible via /loot_monitor_data

Karma — Probe Response Attack
  - Responds to any probe request, claiming the network exists
  - Clients may auto-connect, revealing saved credentials
  - Used with Evil AP

Bruteforce — Credential Cracking
  protocols: ssh (22), ftp (21), telnet (23), smb (445), sql (3306), rdp (3389)
  dictionaries: data/input/dictionary/users.txt, passwords.txt
  output: data/output/crackedpwd/{protocol}.csv
  trigger: /trigger_bruteforce with optional {"protocol": "ssh"|"all"}

NetworkScan — Host Discovery
  tool: nmap
  aggressivity: config "nmap_scan_aggressivity" (-T2 default)
  port range: config portstart (1) to portend (65535 recommended)
  blacklist: mac_scan_blacklist, ip_scan_blacklist
  outputs: data/output/scan_results/result_{timestamp}.csv

VulnerabilityScan — Nmap Vuln Scan
  tool: nmap --script vuln
  interval: config "scan_vuln_interval" (900s default)
  output: data/output/vulnerabilities/

Steal Files — Remote File Extraction
  steal_file_names: ["ssh.csv", "hack.txt"]
  steal_file_extensions: [".your-pax", ".hack", ".flag"]
  output: data/output/data_stolen/

8. ORCHESTRATOR CYCLE
================================================================================

The orchestrator (orchestrator.py) runs continuously after boot:

  LOOP:
    1. Read current_data from CSV (netkb.csv)
    2. For each alive IP and each action:
       - Check if target has the action's required port open
       - Check if parent action succeeded (if child action)
       - Check retry delays (success/failure)
       - Execute action via action.execute(ip, port, row, action_key)
       - Write success/failed_{timestamp} to CSV
    3. If no action executed:
       - Run network scan
       - Re-check targets
       - If vuln scanning enabled: run NmapVulnScanner
       - Run standalone actions (WiFi attacks, Evil AP, etc.)
       - Enter IDLE state for scan_interval seconds
    4. If triggers set (manual mode):
       - manual_scan_trigger → scan
       - manual_bruteforce_trigger → bruteforce (protocol-specific or all)
       - manual_vulnscan_trigger → vulnerability scan
       - manual_steal_trigger → steal files

Manual mode (config: manual_mode=true):
  - Orchestrator enters "triggered" mode (run_triggered)
  - Waits for manual triggers via threading.Event()
  - Each trigger executes the specific action once
  - No autonomous cycling
